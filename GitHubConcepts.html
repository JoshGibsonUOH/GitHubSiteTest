Git Concepts

When beginning to use both GIT and GitHub it is likely that you will come across several new terms that you haven’t heard of before. This section will provide an explanation of some of these terms.

It is not comprehensive, and there may be some others you come across as you use GIT in the future – but these should be all you need to get started.

Repository

A repository (often called a repo for short) is a storage space where your project is kept. Repositories differ from normal files and folders due to the fact that they do not only store your files, but they also store important information such as versions, file histories and other relevant data.

Due to this structure repositories allow you to do several things that normal file paths do not, including working collaboratively with others on the same project, tracking changes you (or someone else!) has made to a project in detail, or reverting a project back to a previous state if something has gone wrong.

Repositories can be stored locally on a machine or storage device, or they can be remote using popular services such as GitHub.

Commit(s)

A commit in GIT is the saved status of a project at a given time. Similar to a checkpoint in a video game when you complete a commit you are creating something that you can go back to, allowing you to return your project in that exact state should you need to.

GIT commits are usually accompanied by a message when they are created, and it is incredibly important that these commit messages are informative and clear – this will help not only you to understand your commits if you need to revisit them in the future, but are also incredibly valuable to other members of your team.

“Added a selection of tests that check the add function to ensure that it returns the correct result” is a lot clearer and easier to understand than “Added tests”.

Remote Repositories

A remote repository is a repository that is hosted somewhere that can be accessed over a network. This is normally on a service like GitHub or GitLab, but creating a remote repository locally or hosting one on a local server is also possible.

Branching

Branching is one of the key aspects of the source control process used by GIT (with the other being merging).

When first adding your project to a GIT repository you are creating your “Main” branch, which is where the stable version of your project exists, meaning that if you were to release your project it would be this “Main” branch that would be released.

Branching gives you the ability to create other “branches” based on this branch that you can work on simultaneously. This opens the door to several different things, including allowing teams to collaboratively work on features together, working on new or experimental features without worrying about damaging your core code base or even working on and testing bug fixes.

All of this is done without changing the “Main” branch, which means that if something goes wrong, your main project is safe.

When a branch has been completed, tested, and you are happy that it can become part of your main project, you can then add it to your “Main” branch – this is known as merging.

[A note on the “Main“ branch: previously, the “Main” branch was referred to as the “Master” branch, so when looking at material to do with GIT and GitHub you may see this naming convention come up. These names are interchangeable and effectively do the same thing. You should however use “Main” going forward, as it is the new convention for GIT repositories]

Merging

Merging is the second key aspect of the source control process implemented by GIT

When a branch has been checked and signed off it can then be merged back into the “Main” branch. This takes all the information from your created branch (including things such as commit histories) and adds them into your main branch, making them part of the main program.

Conflicts

GIT can handle the merging process automatically most of the time, however issues can occur when changes have been made to the same part of the program in two different branches (normally by two different people) this results in something called a conflict.

When there is a conflict GIT can’t automatically decide which changes should take precedence over another so will not merge the branches until the conflicts have been manually removed. Services like GitHub provide a UI to allow you to do this a bit easier.

It is important, especially when working within a team, that you thoroughly check what has been changed before removing conflicts, as it could have unexpected consequences somewhere else in the codebase.

Cloning

Cloning a repository is the process of creating a copy locally of a repository that is located remotely.

You will most often find yourself doing this when you are cloning a repository from GitHub or other similar services to work on locally on your machine.

Fetching and Pulling

Both Fetching and pulling serve similar purposes but have one clear difference that you need to be aware of.

Fetching will pull down the information about any changes in your remote repository (E.G GitHub) and tell you about them. This is useful for if you want to know at a glance if anything has changed.

Fetching, however, will not make any changes to your local repository or branches.

Pulling on the other hand is like completing a fetch and then immediately completing a merge straight after. This will make changes to your local repository in the form of updating the branch you are currently working on to include the changes in the corresponding remote version.

Push

A push is when you upload the changes you have made from your local repository up to a remote repository e.g. uploading changes to a repository hosted on GitHub.

Stashing

In Git a stash is a temporary box where you can place progress or changes that you have made when you don’t necessarily want to commit them. This could be for a variety of reasons, if they are incomplete or if you need to switch branches for example. It may also be that you have an idea that you may want to return to later but is not yet ready for development.

Like when committing to a repository it is important that when creating a stash, you include a clear message explaining what is included in the stash, so if you need to revisit it later you can easily identify what part of the project was being worked on in each stash.
